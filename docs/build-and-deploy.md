# Build And Deploy

This guide covers local builds and web deployment notes.

> Status: local web builds are the current target. Desktop packaging comes later.

## Local Development

From the repository root:

```powershell
cd scratch-gui
npm install
npm start
```

Open:

```text
http://127.0.0.1:8602/editor.html
```

## Local Build

```powershell
cd scratch-gui
npm run build
```

The build output is generated by the `scratch-gui` build process.

## Cloud Builds

Cloud builds are stricter than local builds because they start from a clean install.

If local builds work but Cloudflare or another CI builder fails, suspect:

- dependency missing from `package.json`
- generated files not committed
- workspace package resolution differences
- case-sensitive path differences
- build command running from the wrong folder
- package lock mismatch

## Cloudflare Pages Direction

Use `scratch-gui` as the web build target.

Typical settings:

```text
Root directory: scratch-gui
Build command: npm run build
Build output: build
```

If the output folder differs in the current config, use the folder produced by the build.

## Known Build Problems

### Missing `scratch-sb1-converter`

This usually means a dependency expected by the build is not installed in the cloud environment.

Fix direction:

- make sure it is declared in the correct package
- commit lockfile updates
- avoid relying on local `node_modules`

### Missing `@turbowarp/json`

This means the VM/runtime dependency is not available in the cloud install.

Fix direction:

- declare the dependency in the package that needs it
- commit package and lockfile changes
- verify with a clean install when possible

### Local works but cloud fails

Local `node_modules` can hide missing dependencies.

Useful test:

```powershell
cd scratch-gui
npm run build
```

For a stronger test, use a clean clone or delete/reinstall dependencies when disk space allows.

## Generated Build Outputs

Some packages may require generated outputs or bundled files to exist for deployment.

If a deploy fails with missing block files or generated Scratch Blocks outputs:

- check whether the generated file is ignored
- check whether the build creates it before it is needed
- commit required deploy-time artifacts only when necessary

## Desktop Packaging

Desktop is not the current focus.

The intended direction is to use the existing TurboWarp Desktop-style packaging path in `desktop`, not switch to a new native stack without a reason.

Before desktop work:

- stabilize the web editor
- stabilize `.amx`
- stabilize model import/render
- decide how local file access should work
- review security implications for Network and AmethystAI features

## Release Checklist

Before calling a web release usable:

- `npm run build` passes
- editor opens
- 3D stage renders
- model import works with simple GLB
- motion XYZ blocks work
- camera blocks work
- environment blocks work
- save/load tested
- docs updated
- credits/legal page reachable

## Troubleshooting

### Port already in use

Use the URL printed by the dev server.

### Build is slow

The editor is large. Avoid rebuilding repeatedly for docs-only changes.

### Browser shows old code

Hard refresh. Block definitions and editor bundles can be cached during development.
