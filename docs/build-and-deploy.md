# Build And Deploy

This guide covers local web builds, Cloudflare Pages deployment notes, and the current Electron desktop build.

> Status: web builds and 64-bit Electron desktop builds are active. Desktop export is the current paid-binary differentiator. 32-bit Electron and Tauri x64 are planned/experimental, not stable release targets.

## Local Development

From the repository root:

```powershell
cd scratch-gui
npm install
npm start
```

Open:

```text
http://127.0.0.1:8602/editor.html
```

## Local Build

```powershell
cd scratch-gui
npm run build
```

The build output is generated by the `scratch-gui` build process.

## Desktop Build

From the repository root:

```powershell
cd desktop
npm install
npm run fetch
npm run webpack:compile
npm run electron:package:dir
```

The unpacked Windows test build is created at:

```text
desktop\dist\win-unpacked\Amethyst.exe
```

For a production-ish renderer bundle, run:

```powershell
cd desktop
npm run webpack:prod
npm run electron:package:dir
```

The current packaged app target is 64-bit Electron. The product direction also includes 32-bit Electron and experimental Tauri x64 later, but do not call those release-ready until they have their own build scripts, test pass, and packaging docs.

## Cloud Builds

Cloud builds are stricter than local builds because they start from a clean install.

If local builds work but Cloudflare or another CI builder fails, suspect:

- dependency missing from `package.json`
- generated files not committed
- workspace package resolution differences
- case-sensitive path differences
- build command running from the wrong folder
- package lock mismatch

## Cloudflare Pages Direction

Use `scratch-gui` as the web build target.

Typical settings:

```text
Root directory: scratch-gui
Build command: npm run build
Build output: build
```

If the output folder differs in the current config, use the folder produced by the build.

## Known Build Problems

### Missing `scratch-sb1-converter`

This usually means a dependency expected by the build is not installed in the cloud environment.

Fix direction:

- make sure it is declared in the correct package
- commit lockfile updates
- avoid relying on local `node_modules`

### Missing `@turbowarp/json`

This means the VM/runtime dependency is not available in the cloud install.

Fix direction:

- declare the dependency in the package that needs it
- commit package and lockfile changes
- verify with a clean install when possible

### Local works but cloud fails

Local `node_modules` can hide missing dependencies.

Useful test:

```powershell
cd scratch-gui
npm run build
```

For a stronger test, use a clean clone or delete/reinstall dependencies when disk space allows.

## Generated Build Outputs

Some packages may require generated outputs or bundled files to exist for deployment.

If a deploy fails with missing block files or generated Scratch Blocks outputs:

- check whether the generated file is ignored
- check whether the build creates it before it is needed
- commit required deploy-time artifacts only when necessary

## Desktop HTML Export

The desktop app has the Amethyst single-file HTML export path. Web builds should not expose this as a normal feature.

Export direction:

- input project format: `.amx`
- output format: one offline `.html` file
- player runtime: bundled into the exported HTML by the desktop app
- project payload: embedded as base85 chunks, not one huge base64 string
- goal: users edit in Amethyst, then share/open the exported HTML for playback

This exporter intentionally follows the smarter TurboWarp Packager pattern: avoid giant JavaScript strings, avoid one giant base64 blob, and decode project bytes from chunks.

Before treating desktop export as release-ready for users:

- test a tiny `.amx`
- test a project with several scripts
- test a project with model actors
- test a project with camera/environment blocks
- test a larger model-heavy project
- open the exported HTML with internet off
- verify no `.sb3` wording appears in the export UI
- verify Network and AmethystAI features do not silently create unsafe desktop-only behavior

## Release Checklist

Before calling a web release usable:

- `npm run build` passes
- editor opens
- 3D stage renders
- model import works with simple GLB
- motion XYZ blocks work
- camera blocks work
- environment blocks work
- save/load tested
- docs updated
- credits/legal page reachable

Before calling a desktop release usable:

- `npm run webpack:compile` or `npm run webpack:prod` passes in `desktop`
- `npm run electron:package:dir` passes
- `desktop\dist\win-unpacked\Amethyst.exe` opens
- `.amx` save/load works in the desktop app
- single-file HTML export creates a file
- exported HTML opens offline
- exported HTML loads the project without requiring refresh
- model actors, camera, and environment state survive export
- paid-binary notice in docs is accurate

## Troubleshooting

### Port already in use

Use the URL printed by the dev server.

### Build is slow

The editor is large. Avoid rebuilding repeatedly for docs-only changes.

### Browser shows old code

Hard refresh. Block definitions and editor bundles can be cached during development.
